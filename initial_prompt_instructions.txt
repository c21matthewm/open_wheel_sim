Simple Arcade Race Sim Prompt Initial Instructions


You are building a lightweight native macOS sim-racing prototype focused on low latency, small build size, Logitech G29 wheel/pedal support, believable open-wheel vehicle physics, tunable force feedback where macOS allows it, and fast performance on a Mac connected to a TV.
Project name: Lightweight Open-Wheel Sim Prototype
Primary target:
* macOS desktop application.
* Runs locally on the Mac, not in a browser.
* Displayed on a TV or monitor over HDMI, USB-C-to-HDMI, or DisplayPort-to-HDMI.
* Target render resolutions:
    * 854x480
    * 960x540
    * 1280x720
* Must support fullscreen and borderless windowed mode.
* Target frame rate: stable 60 FPS minimum.
* Target physics rate: 120 Hz preferred, 240 Hz only if stable.
* Target input latency: as low and consistent as possible.
* Prioritize deterministic physics, clean architecture, small assets, and no unnecessary engine bloat.
* Do not use Unity, Unreal, or Godot for the first version.
Important macOS-specific constraint:The Logitech Steering Wheel SDK used on Windows should not be treated as available on macOS. On macOS, wheel input and force feedback must be implemented through a platform abstraction. Use SDL3 joystick APIs and/or Apple Game Controller framework for device input. Investigate SDL3 haptic support, Apple controller haptics, IOKit HID, or other macOS-compatible paths for force feedback, but do not assume Windows DirectInput or Logitech’s Windows SDK exists on macOS.
Preferred stack:
* Language: C++20 or C++23.
* Compiler: Apple Clang via Xcode Command Line Tools.
* Build system: CMake.
* IDE/editor compatibility:
    * Xcode
    * VS Code
    * Cursor
    * CLion if available
* Window/input/audio layer:
    * SDL3 for windowing, keyboard, mouse, joystick, gamepad, timing, and audio.
* Rendering:
    * Prefer Metal as the primary renderer.
    * Acceptable approaches:
        * Native Metal renderer using Objective-C++ bridge files where needed.
        * SDL3 window + Metal layer.
        * SDL3 GPU API if stable and practical for the project.
    * Avoid relying on OpenGL as the primary renderer because OpenGL is deprecated on macOS.
* UI/debug overlay:
    * Dear ImGui if it integrates cleanly with Metal/SDL3.
    * Otherwise, implement a minimal custom debug overlay.
* Config format:
    * JSON, TOML, or INI.
    * Prefer TOML or JSON for readability.
* Asset formats:
    * glTF or OBJ for simple meshes.
    * PNG or WebP for small textures.
    * WAV or OGG for audio.
    * Keep all assets small, replaceable, and unlicensed.
* Version control:
    * Git.
    * GitHub or GitLab.
    * Use Git LFS only if asset files become large.
Core goal:Create a playable single-car, single-track racing prototype with believable open-wheel/simcade physics and tunable steering, braking, grip, and force-feedback behavior. The prototype should be simple in content but not crude in feel.
The game should feel closer to a lightweight Indy-style open-wheel sim than an arcade racer, but the scope should stay small enough for one developer or a coding agent to build iteratively.

Updated “how to run it” instructions for macOS
You still do not need to host the game somewhere for it to run. Hosting would add latency and would not help with wheel force feedback.
The workflow should be:

Mac local folder → GitHub repo for backup/version control → local CMake build → macOS app → HDMI/USB-C-to-HDMI → TV

Use GitHub for the code. Use GitHub Releases later for downloadable .zip or .dmg builds. GitHub Releases are intended for packaging project versions with downloadable release assets, and Git LFS is useful if your repo starts accumulating larger binary assets.
Recommended macOS development setup
Use this:

Operating system: macOS
Compiler/toolchain: Xcode Command Line Tools
Build system: CMake
Editor/IDE: VS Code, Cursor, CLion, or Xcode
Window/input/audio: SDL3
Renderer: Metal
Version control: Git + GitHub
Wheel software: Logitech G HUB only if useful for device recognition/configuration

You can work in VS Code or Cursor and build from the terminal. You do not have to work inside Xcode, but you should install Xcode Command Line Tools because they provide Apple Clang, build tools, headers, and SDK access.
Typical setup:

xcode-select --install
brew install cmake sdl3
git clone <your-repo-url>
cd sim-racing-prototype
cmake -S . -B build
cmake --build build --config Release

What changes most from the Windows version
The Windows prompt could reasonably say:

Use Logitech Steering Wheel SDK for G29 input and force feedback.

The macOS version should say:

Use SDL3 and/or Apple Game Controller framework for wheel input.
Treat force feedback as an experimental backend.
Do not depend on Logitech’s Windows SDK.

That is the most important correction. Steering, throttle, brake, and paddles are achievable first. High-quality G29 force feedback on macOS is the risk area.
TV display setup on macOS
Use:

Mac → HDMI or USB-C-to-HDMI → TV

Then:

System Settings → Displays → choose TV → set resolution/refresh rate

For lowest display lag:
* Enable Game Mode on the TV.
* Use fullscreen or borderless fullscreen.
* Use 60 Hz output initially.
* Avoid AirPlay for this project.
* Use a wired USB connection for the G29, not Bluetooth or adapters through a flaky hub.
